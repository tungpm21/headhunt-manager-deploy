## Đánh giá phân tích của Codex

### ✅ Đúng (4/5 claims)

#### 1. `getPublicJobs()` chạy 5 query song song — ĐÚNG
- **Vị trí:** `src/lib/public-actions.ts:239-281`
- Thực tế: 5 queries trong `Promise.all` + thêm 1 raw `unnest` query nếu có keyword
- Với keyword search: **6 queries mỗi request** là chính xác
- Unnest query không có index hỗ trợ → full table scan khi data lớn

#### 2. Double-fetch ở slug pages — ĐÚNG (nghiêm trọng hơn Codex nói)
- **Vị trí:** `viec-lam/[slug]/page.tsx:28` và `:47`
- `generateMetadata()` gọi `getPublicJobBySlug(slug)` → line 28
- Page render lại gọi `getPublicJobBySlug(slug)` → line 47
- Mỗi lần gọi = 2 DB queries + fire `incrementJobPostingView()`
- **Kết quả thực tế: mỗi page view = 4 DB queries + 2 view increment writes**
- Đây là bug quan trọng nhất, Codex đúng

#### 3. `incrementJobPostingView()` là write trong read path — ĐÚNG
- Gọi fire-and-forget (không await) nên không block TTFB
- Nhưng tạo thêm DB write connection cho mỗi request
- Bị gọi **2 lần** do double-fetch ở trên

#### 4. `setTimeout` không cleanup trong `candidate-filters.tsx` — ĐÚNG
- **Dòng 102:** `setTimeout(() => update({ search: value || null }), 300)`
- **Dòng 245:** `setTimeout(() => update({ skills: value || null }), 400)`
- Không có cleanup → gõ nhanh 5 ký tự = 5 `router.push()` liên tiếp
- Gây re-render cascade + 5 lần full page navigation

### ⚠️ SAI / Chưa chính xác (1/5 claims)

#### 5. `expireSubscriptionsIfNeeded()` chặn read path — **SAI QUAN TRỌNG**
- Codex nói: "buộc dynamic và phải chạm DB mỗi lần"
- **Thực tế code `subscriptions.ts:3-16`:**
  ```typescript
  const SUBSCRIPTION_SYNC_INTERVAL_MS = 60_000; // 60 giây
  let lastSubscriptionSyncAt = 0;

  if (!force && now - lastSubscriptionSyncAt < SUBSCRIPTION_SYNC_INTERVAL_MS) {
    return 0; // Skip — trả về ngay
  }
  ```
- Hàm đã có **in-memory throttle 60 giây**: sau lần đầu, 59 giây tiếp theo không chạm DB
- Trong môi trường Vercel serverless (cold start per request), `lastSubscriptionSyncAt` reset mỗi cold start
- **Cold start issue = thật**, nhưng warm instances thì không chặn
- Đây không phải nguyên nhân chính của TTFB ~3-5s

## 🔴 Critical Issues — Phải sửa ngay

### BUG-01: Double-fetch trong slug pages
| | Chi tiết |
|--|--|
| **File** | `viec-lam/[slug]/page.tsx`, `cong-ty/[slug]/page.tsx` |
| **Hậu quả** | Mỗi page view tốn gấp đôi DB queries |
| **Fix** | Cache data với `cache()` từ React hoặc dùng `unstable_cache` |

```typescript
// Fix: Wrap with React cache() để deduplicate trong cùng 1 request
import { cache } from "react";
const getCachedJobBySlug = cache(getPublicJobBySlug);
// Cả generateMetadata() và page() dùng cùng 1 cached result
```

### BUG-02: Debounce không cleanup trong candidate-filters
| | Chi tiết |
|--|--|
| **File** | `candidate-filters.tsx:100-103` và `243-246` |
| **Hậu quả** | Gõ nhanh → nhiều navigation cùng lúc → UI lag |
| **Fix** | Dùng `useRef` để cleanup setTimeout trước khi set mới |

```typescript
// Fix: proper debounce
const timerRef = useRef<ReturnType<typeof setTimeout>>();
onChange={(e) => {
  clearTimeout(timerRef.current);
  timerRef.current = setTimeout(() => update({ search: e.target.value || null }), 300);
}}
```

## 🟡 Warnings — Nên sửa

### WARN-01: `getPublicJobs()` filter queries không cached
- `distinctIndustries`, `distinctLocations`, `distinctWorkTypes` chạy mỗi request
- Dữ liệu này thay đổi rất ít (cùng ngành/khu vực)
- **Fix:** Wrap với `unstable_cache` + revalidate 1 giờ

### WARN-02: Unnest raw query thiếu index
- `findJobPostingIdsBySkillKeyword()` dùng `unnest("skills")` full scan
- **Fix:** Tạo GIN index trên column `skills[]` trong Prisma migration

### WARN-03: `expireSubscriptionsIfNeeded()` reset trên cold start
- Vercel serverless → mỗi cold start reset in-memory counter
- **Fix:** Dùng `unstable_cache` với revalidate 60s thay vì in-memory variable
